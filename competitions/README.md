# 比赛工作区

用于存放实际参赛时的代码、数据和论文。

## 📁 组织结构

按年份组织，每次比赛创建独立目录：

```
2026/
├── problem_analysis/      # 问题分析文档
│   ├── problem.pdf       # 题目原文
│   ├── analysis.md       # 问题分析笔记
│   └── ideas.md          # 建模思路讨论
│
├── code/                 # 比赛代码
│   ├── question1.py      # 按问题组织
│   ├── question2.py
│   ├── utils.py          # 工具函数
│   └── config.py         # 配置文件
│
├── data/                 # 比赛数据
│   ├── raw/              # 原始数据
│   ├── processed/        # 处理后数据
│   └── results/          # 计算结果
│
├── paper/                # 论文撰写
│   ├── main.tex          # LaTeX主文件
│   ├── sections/         # 各章节
│   ├── figures/          # 图片
│   └── references.bib    # 参考文献
│
└── results/              # 最终结果
    ├── figures/          # 最终图表
    ├── tables/           # 数据表格
    └── final_paper.pdf   # 提交的论文
```

## ⏰ 时间管理建议

### Day 1（选题日）
- [ ] 7:00-9:30 各自读题记录思路
- [ ] 9:30-10:00 讨论排除无思路题目
- [ ] 10:00-12:00 查资料确定选题
- [ ] 14:00-18:00 确定方案、大纲、数据来源
- [ ] 18:00-22:00 完成第一问，写引言

### Day 2（攻坚日）
- [ ] 建模手：完成前三问建模
- [ ] 编程手：完成前两问求解
- [ ] 论文手：撰写前两问正文

### Day 3（冲刺日）
- [ ] 完成所有问题
- [ ] 前三问中文稿完成

### Day 4（打磨日）
- [ ] 上午：完成整体中文稿
- [ ] 下午：翻译、排版、润色

### Day 5（提交日）
- [ ] 8:00-10:00 交叉检查提交

## 💡 工作建议

1. **版本控制**：及时提交代码到git，避免丢失
2. **定期备份**：论文和代码多处备份
3. **清晰命名**：文件名包含日期和版本号
4. **注释充分**：代码要有详细注释，方便回查
5. **团队协作**：使用共享文档实时同步进度

## 🚨 注意事项

- 确保网络畅通，可能需要下载数据
- 提前测试LaTeX编译环境
- 准备好翻译工具（DeepL、Google Translate等）
- 论文提交前仔细检查格式和语法
- 记住提交截止时间（比赛开始后96小时）
